# Miles Theme Audit — Documentation Overview

## Purpose

This documentation tracks the iterative build and review process for a WordPress block theme generated by [Miles AI](https://bymiles.ai). The goal is to create a feedback loop: prompt Miles, export the result, review the output, and document what's good, what's missing, and what needs to change. Over time this becomes a playbook that helps Miles produce better themes out of the box.

## Process

1. **Prompt Miles** — Give Miles a specific instruction (e.g. "add a contact page template" or "set up the color palette").
2. **Export the theme** — Download the generated theme files.
3. **Review with Claude** — Drop the exported files here and ask for a structured review.
4. **Document findings** — A review file is created at `/docs/prompts/prompt-review-XX.md` covering what was generated, what's correct, what's missing, and what needs fixing.
5. **Feed back to Miles** — Use the findings to refine future prompts or to improve the Miles agent itself.

### Folder structure

```
docs/
├── OVERVIEW.md
└── prompts/
    ├── prompt-review-01.md
    ├── prompt-review-02.md
    └── ...
```

Each review file follows a consistent format (see Review Template below).

## What We're Looking For

### theme.json — Design System Completeness

| Area | What to check | Why it matters |
|------|---------------|----------------|
| **Colors** | Are palette slugs semantic (e.g. `primary`, `secondary`) or arbitrary? Are hover/focus/active states covered? Is there a white or light background option? | Users need a predictable, complete palette in the Site Editor. Missing colors force inline styles. |
| **Typography** | Are font sizes fluid (`clamp`)? Do slug names follow WordPress conventions (`small`, `medium`, `large`, `x-large`)? Are heading-level sizes defined? | Non-standard slug names won't map to the editor's size picker. Static sizes break on mobile. |
| **Spacing** | Is the scale consistent (e.g. 4px base unit)? Are there enough steps? Are slug names standard (`20`–`80`)? | Gaps in the scale force authors to hardcode values. |
| **Shadows** | Are presets actually used in templates/patterns? Do they work on the dark background? | Unused presets are clutter. Shadows designed for light backgrounds disappear on dark ones. |
| **Gradients** | Do slugs have meaningful names? Are gradients used consistently? | `extracted-linear-1` tells an author nothing. |
| **Layout** | Is `contentSize` appropriate? Is `wideSize` differentiated enough from content? | Too-close values make "wide" meaningless. |
| **Custom properties** | Are line-height and font-weight tokens actually referenced? Could they be replaced by standard settings? | Custom tokens that duplicate built-in settings add complexity without benefit. |
| **Block styles** | Are core block overrides intentional and complete? Do resets (padding: 0, margin: 0) break expected behavior? | Aggressive resets can cause layout issues when authors add standard blocks. |

### Templates & Patterns

| Area | What to check |
|------|---------------|
| **Coverage** | Are all standard templates present (front-page, page, single, archive, search, 404, index, home)? |
| **Consistency** | Do all templates use the same header/footer parts? Is spacing consistent? |
| **Hardcoded content** | Are there strings that should be dynamic (site title, tagline, copyright year)? |
| **Accessibility** | Proper heading hierarchy? Skip links? Focus states? Color contrast? |
| **Semantic HTML** | Correct use of `<main>`, `<article>`, `<nav>`, `<aside>`? |

### CSS & JavaScript

| Area | What to check |
|------|---------------|
| **Hardcoded values** | Colors, fonts, or spacing in CSS that should reference theme.json tokens |
| **Aggressive resets** | Global `* { margin: 0; padding: 0; }` can break core block styles |
| **Duplicate declarations** | CSS that re-declares what theme.json already handles (body font, background color) |
| **JS dependencies** | Does the JS duplicate what the Interactivity API could handle? |
| **Editor parity** | Do front-end-only styles cause a mismatch between the editor and the published site? |

### functions.php

| Area | What to check |
|------|---------------|
| **Enqueue patterns** | Correct use of `wp_enqueue_style` / `wp_enqueue_scripts` with versioning |
| **Theme supports** | Are all necessary supports declared? Any redundant ones for FSE themes? |
| **Text domain** | Consistent and matching `style.css` |

## Suggested Prompts for Future Reviews

Use these prompts (or variations) with Miles to systematically build out the theme. Each prompt becomes a new `prompt-review-XX.md` file.

### Foundation prompts
- "Set up the base color palette and typography scale"
- "Create the header and footer template parts"
- "Build the front page template"
- "Build standard inner page templates (page, single, archive, search, 404)"

### Content prompts
- "Add a blog query loop with pagination"
- "Create a contact page pattern"
- "Add a services/features section pattern"
- "Build a testimonials or reviews pattern"
- "Create an image gallery or portfolio grid"

### Refinement prompts
- "Add a light color style variation"
- "Ensure all font sizes use fluid clamp values"
- "Rename gradient slugs to be descriptive"
- "Add proper focus and hover states to all interactive elements"
- "Audit the theme for WCAG AA color contrast"

### Advanced prompts
- "Add a custom block style for the quote block"
- "Create a style variation with an alternate font pairing"
- "Optimize the spacing scale to follow a consistent ratio"
- "Replace scroll animation JS with the Interactivity API"

## Review Template

Each `prompt-review-XX.md` should follow this structure:

```markdown
# Prompt XX — [Short description]

## Prompt Given
> The exact prompt sent to Miles.

## What Was Generated
Summary of files added or changed.

## What's Good
- Bullets on what works well.

## What's Missing
- Bullets on gaps, omissions, or incomplete implementations.

## Issues & Concerns
- Bullets on problems, inconsistencies, or things that will cause trouble.

## Recommendations
- Bullets on specific changes or improvements.
```
